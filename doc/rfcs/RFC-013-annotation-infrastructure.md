# RFC-013: Annotation Infrastructure — Storage, Resolution, and Editor Integration

**Status:** Proposed
**Date:** 2026-03-23
**Authors:** Anne Schuth

## Context

RFC-005 defines the annotation *format*: W3C Web Annotation Data Model with TextQuoteSelector for version-resilient text anchoring. It specifies what an annotation looks like (selector, motivation, body, resolution states) but is explicitly storage-agnostic. It does not say where annotations live, how they're loaded, who creates them, or how they're displayed.

This RFC addresses the implementation questions that RFC-005 leaves open:

1. **Working annotations** — needed for the editor to bridge the gap between law text and `machine_readable` execution logic.
2. **Conflict resolution for generated annotations** — when annotations are generated by tooling (e.g., LLM-produced linking annotations), how are conflicts between generated and human-authored annotations handled?
3. **Responsible law-to-YAML mapping** — linking annotations connect specific text fragments to the corresponding `machine_readable` elements, making the interpretation chain explicit and auditable.
4. **Annotation types and storage locations** — different annotation types serve different purposes and may live in different places.
5. **Annotation history** — version tracking via Git.
6. **Annotation scope** — what does an annotation relate to? The entire law across all versions? A specific version? Is it personal or public? Does it target structure or content?

Two other RFCs shape the annotation infrastructure:

- **RFC-009** (Multi-Organisation Execution) introduces `competent_authority` (*bevoegd gezag*) as the boundary that determines who can authoritatively execute what. The same concept applies to annotations: the competent authority's (*bevoegd gezag*) annotation linking text to `machine_readable` represents the official interpretation (*gezaghebbende interpretatie*). Other organisations, legal experts, and citizens can bring their own annotations alongside — as advisory perspectives, not as the authoritative reading.

- **RFC-010** (Federated Corpus) introduces a "bring your own regulations" model where municipalities (*gemeenten*), provinces (*provincies*), and other organisations maintain laws in their own repositories. Annotations need the same federated model: any organisation can annotate any law from their own perspective, in their own repository.

### Current state

Nothing is implemented on `main`. A historical Python proof-of-concept exists on the `feature/annotation-resolver` branch (TextQuoteSelector resolution, fuzzy matching, BDD tests), but the stack has moved to Rust (engine) and Vue 3 (frontend). The Python code will not be reused; the BDD scenarios will be ported.

## Decision

Eight interconnected decisions form the annotation infrastructure.

### 1. Storage: Sidecar YAML files

Annotations are stored in separate YAML files alongside law files, not embedded in the law YAML. This preserves the verbatim legal text (RFC-005 requirement: annotations must not modify the source) and enables independent versioning.

**Directory convention:**

```
corpus/
  regulation/nl/
    wet/
      wet_op_de_zorgtoeslag/
        2025-01-01.yaml                        # law text (unchanged)
  annotations/
    wet_op_de_zorgtoeslag/
      annotations.yaml                          # law-level annotations (all versions)
      2025-01-01.annotations.yaml               # version-specific annotations (optional)
```

- `annotations.yaml` — law-level annotations that apply across all versions. The TextQuoteSelector resolves on whichever version is being viewed.
- `{valid_from}.annotations.yaml` — annotations specific to a particular law version. Used when the annotated text only exists in that version (e.g., a percentage that was changed in a later amendment).

**Discovery:** convention-based. Given a law with `$id: zorgtoeslagwet`, the engine looks for annotations at `corpus/annotations/zorgtoeslagwet/annotations.yaml`. No registry manifest is needed for local annotations.

**Validation:** annotation files conform to `schema/v0.4.0/annotation-schema.json`, validated by `just validate-annotations`.

**Example annotation file:**

```yaml
# corpus/annotations/wet_op_de_zorgtoeslag/annotations.yaml
$schema: "https://raw.githubusercontent.com/MinBZK/regelrecht-mvp/refs/heads/main/schema/v0.4.0/annotation-schema.json"
annotations:
  - type: Annotation
    motivation: linking
    creator: "Dienst Toeslagen"
    target:
      source: "regelrecht://zorgtoeslagwet"
      selector:
        type: TextQuoteSelector
        exact: "zorgtoeslag ter grootte van dat verschil"
        prefix: "heeft de verzekerde aanspraak op een "
        suffix: ""
        regelrecht:hint:
          type: CssSelector
          value: "article[number='2']"
          refinedBy:
            type: TextPositionSelector
            start: 112
            end: 152
    body:
      type: SpecificResource
      source: "regelrecht://zorgtoeslagwet/artikel_2#hoogte_zorgtoeslag"
      purpose: linking
    resolution: found
    workflow: resolved
```

### 2. Federated annotation sources

Annotations follow the same "bring your own" pattern as laws in RFC-010. Any organisation can maintain annotations in their own repository, alongside their regulations or in a dedicated annotation repository.

**Required structure in a source repository:**

```
annotations/
  {law_id}/
    annotations.yaml
```

When a source in `corpus-registry.yaml` (RFC-010) includes an `annotations/` directory, the engine discovers and loads those annotations alongside the source's regulations.

**Example: municipality (*gemeente*) annotating a national law:**

```
# In gemeente-amsterdam/regelrecht-amsterdam repo:
regulation/nl/
  verordening/
    afstemmingsverordening/
      2025-01-01.yaml                           # Amsterdam's ordinance
annotations/
  wet_op_de_zorgtoeslag/
    annotations.yaml                             # Amsterdam's perspective on national law
  afstemmingsverordening/
    annotations.yaml                             # annotations on their own ordinance
```

Amsterdam's annotation on the Healthcare Allowance Act (*Wet op de zorgtoeslag*) might link text about "toeslagpartner" to their local implementation of the partner check, or add a comment explaining how Amsterdam interprets a specific provision in the context of their municipal social assistance (*bijstand*) policy.

**Personal annotations:** a `.local.annotations.yaml` file (gitignored) follows the RFC-010 `.local.yaml` override pattern. Personal annotations are loaded alongside public annotations but are not committed to the repository.

**Loading order:** the engine loads annotations from all registered sources. Annotations from different sources are **layered**, not prioritised (see Decision 7).

### 3. Annotation authority and provenance

Each annotation carries a `creator` field (part of the W3C Web Annotation Data Model) identifying who created it. The relationship between the creator and the annotated law determines the annotation's authority level.

| Creator | Relationship to `competent_authority` | Authority level | Example |
|---------|--------------------------------------|----------------|---------|
| Competent authority (*bevoegd gezag*) | Same org | Authoritative (*gezaghebbend*) | Allowances Service (*Dienst Toeslagen*) annotates Healthcare Allowance Act (*Wet op de zorgtoeslag*) |
| Other government org | Different org | Advisory (*adviserend*) | Municipality (*gemeente*) annotates national law from local perspective |
| Legal expert / researcher | No org | Personal (*persoonlijk*) | Law professor adds explanatory comment |
| Automated tooling | Generated | Generated (*gegenereerd*) | LLM-produced linking annotation |

The authority level is **derived at display time**, not stored. It follows the same principle as RFC-009's execute/accept boundary: the law determines authority, the engine derives behaviour.

**Derivation logic:**

```
Is the annotation's creator the competent_authority for the target law?
  YES → Authoritative
  NO  →
    Is the creator a government organisation?
      YES → Advisory
      NO  →
        Is the creator an automated tool?
          YES → Generated
          NO  → Personal
```

For the MVP, `creator` is a string field (e.g., `"Dienst Toeslagen"`, `"LLM-generated"`, `"J. de Vries"`). When RFC-009's identity model (`EngineIdentity` with OIN) is implemented, `creator` can carry a structured identity with signature verification. The schema is designed to accommodate both:

```yaml
# MVP: simple string
creator: "Dienst Toeslagen"

# Future: structured identity (W3C Agent)
creator:
  type: Organisation
  name: "Dienst Toeslagen"
  organisation_id: "00000004003214345000"  # OIN
```

**Provenance:** the annotation file's Git history provides when each annotation was created and by whom (`git log`, `git blame`). The annotation schema optionally includes `created` and `modified` timestamps for systems that don't use Git.

### 4. Scope model

Each scope dimension maps to a concrete mechanism:

| Scope dimension | Mechanism | How it works |
|-----------------|-----------|--------------|
| Entire law, all versions (*hele wet, alle versies*) | Law-level `annotations.yaml` | TextQuoteSelector resolves on whichever version is being viewed. An annotation on "zorgtoeslag" finds the word in both the 2020 and 2025 versions. |
| Specific version (*specifieke versie*) | Version-specific `{valid_from}.annotations.yaml` | Annotation on a percentage that was changed by Staatsblad 2008, 516 only makes sense on the pre-2008 version. |
| Personal / public (*persoonlijk / publiek*) | Public: in Git. Personal: `.local.annotations.yaml` (gitignored) | An expert's draft annotations before publishing; a student's study notes. |
| Structure or content (*structuur of inhoud*) | Annotations target *text content* via TextQuoteSelector | The annotation on "zorgtoeslag" finds the word regardless of which article it's in. If article 2 is renumbered to article 3, the annotation follows the text. Article numbers appear only as performance hints (non-authoritative). |

### 5. Annotation types and their use cases

The W3C Web Annotation vocabulary defines 13 motivation types. Four are primary for regelrecht:

#### Linking (*koppeling*)

Connects text to a `machine_readable` element. This is the most critical annotation type — it makes the interpretation chain from law text to executable logic explicit and auditable.

```yaml
- type: Annotation
  motivation: linking
  creator: "Dienst Toeslagen"
  target:
    source: "regelrecht://zorgtoeslagwet"
    selector:
      type: TextQuoteSelector
      exact: "zorgtoeslag ter grootte van dat verschil"
      prefix: "heeft de verzekerde aanspraak op een "
      suffix: ""
  body:
    type: SpecificResource
    source: "regelrecht://zorgtoeslagwet/artikel_2#hoogte_zorgtoeslag"
    purpose: linking
```

The `body.source` uses a `regelrecht://` URI (as defined in `packages/engine/src/uri.rs`) pointing to the specific execution element. The fragment (`#hoogte_zorgtoeslag`) identifies the output name.

#### Commenting (*toelichting*)

Human explanation of a legal concept or provision.

```yaml
- type: Annotation
  motivation: commenting
  creator: "J. de Vries"
  target:
    source: "regelrecht://zorgtoeslagwet"
    selector:
      type: TextQuoteSelector
      exact: "normpremie"
      prefix: "Indien de "
      suffix: " voor een verzekerde"
  body:
    type: TextualBody
    value: >-
      The normative premium (normpremie) is a notional amount that represents
      what the insured person is expected to contribute to health insurance.
      It is based on income, not on the actual premium paid.
    purpose: commenting
    format: text/plain
    language: en
```

#### Tagging (*classificatie*)

Classification of legal concepts for search, analysis, and cross-referencing.

```yaml
- type: Annotation
  motivation: tagging
  creator: "regelrecht-tooling"
  target:
    source: "regelrecht://zorgtoeslagwet"
    selector:
      type: TextQuoteSelector
      exact: "verzekerde"
      prefix: "heeft de "
      suffix: " aanspraak op een zorgtoeslag"
  body:
    type: TextualBody
    value: "legal-subject"
    purpose: tagging
```

#### Questioning (*vraag*)

Open question raised during interpretation. Tracked via the `workflow` field.

```yaml
- type: Annotation
  motivation: questioning
  creator: "interpreter-team"
  target:
    source: "regelrecht://zorgtoeslagwet"
    selector:
      type: TextQuoteSelector
      exact: "drempelinkomen"
      prefix: "procent van het "
      suffix: ", vermeerderd met"
  body:
    type: TextualBody
    value: "Is this the same drempelinkomen defined in Awir article 2?"
    purpose: questioning
  workflow: open
```

### 6. Resolver architecture

The TextQuoteSelector resolver is implemented in Rust as a module within the engine crate. It is exposed to the frontend via WASM.

**Module structure:**

```
packages/engine/src/
  annotation/
    mod.rs           # pub mod types; pub mod resolver;
    types.rs         # Data types
    resolver.rs      # Resolution algorithm
```

**Types** (in `types.rs`):

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TextQuoteSelector {
    pub exact: String,
    pub prefix: Option<String>,
    pub suffix: Option<String>,
    pub hint: Option<SelectorHint>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SelectorHint {
    pub article_number: String,
    pub start: Option<usize>,
    pub end: Option<usize>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MatchResult {
    pub status: MatchStatus,
    pub matches: Vec<TextMatch>,
}

#[derive(Debug, Clone, PartialEq, Serialize, Deserialize)]
pub enum MatchStatus {
    Found,
    Orphaned,
    Ambiguous,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TextMatch {
    pub article_number: String,
    pub start: usize,
    pub end: usize,
    pub confidence: f64,
    pub matched_text: String,
}
```

**Resolution algorithm** (in `resolver.rs`):

```
resolve(selector, articles) -> MatchResult:
  1. Build full text from all articles, tracking article boundaries
  2. Try exact match:
     - Search for prefix + exact + suffix as substring
     - If found: return Found with confidence 1.0
  3. If hint is present:
     - Try exact match within hinted article only
     - If found: return Found
     - If not: continue to full search (hint was outdated)
  4. Fuzzy match:
     - For each sliding window of len(exact) +/- 30% in the full text:
       - exact_score = levenshtein_similarity(exact, window_text)
       - prefix_score = levenshtein_similarity(prefix, text_before_window)
       - suffix_score = levenshtein_similarity(suffix, text_after_window)
       - score = exact_score * 0.5 + prefix_score * 0.25 + suffix_score * 0.25
     - Collect matches above threshold (0.7)
  5. If one match: return Found with confidence = score
  6. If multiple matches with equal score: return Ambiguous
  7. If no matches: return Orphaned
```

**Dependency:** `strsim` crate for Levenshtein distance.

**WASM bindings** (additions to `packages/engine/src/wasm.rs`):

```rust
#[wasm_bindgen]
impl WasmEngine {
    /// Resolve a single TextQuoteSelector against a loaded law
    pub fn resolve_annotation(
        &self,
        law_id: &str,
        selector: JsValue,
    ) -> Result<JsValue, JsValue>;

    /// Resolve all annotations from a YAML string against a loaded law
    pub fn resolve_annotations(
        &self,
        law_id: &str,
        annotations_yaml: &str,
    ) -> Result<JsValue, JsValue>;
}
```

**BDD tests:** port the 8 scenarios from `feature/annotation-resolver:features/annotation.feature`:

1. Exact match — Zorgtoeslagwet 2025 article 4a
2. Article renumbered — Staatsblad 2024, 291 (article 3 -> article 4a)
3. Text change — Staatsblad 2008, 516 (percentage 3,5 -> 2,7)
4. Orphaned annotation — text fully removed
5. Ambiguous match — common word without context
6. Unique with context — prefix/suffix disambiguation
7. Hint optimization — correct hint
8. Hint fallback — outdated hint

### 7. Conflict resolution and layering

Annotations from different sources **layer** — they do not conflict. This is fundamentally different from laws, where RFC-010 uses priority to resolve `$id` collisions. Two organisations can both annotate the same text fragment; both annotations are valid and visible.

**Layering model:**

```
Law text: "heeft de verzekerde aanspraak op een zorgtoeslag ter grootte van dat verschil"
           ─────────────────────────────────────────────
                                    │
           ┌────────────────────────┼─────────────────────────┐
           │                        │                          │
    Dienst Toeslagen         Gemeente Amsterdam          LLM-generated
    motivation: linking      motivation: commenting      motivation: linking
    body: #hoogte_zorg-      body: "This is the core     body: #berekening_
          toeslag                   provision that our          zorgtoeslag
    authority: authoritative       bijstand calculation
                                   references"
                             authority: advisory          authority: generated
```

All three annotations coexist. The editor displays them layered, with authoritative annotations visually distinguished (e.g., solid highlight vs dashed border).

**When law text changes:**

1. The resolver runs all annotations against the new version
2. Annotations that still match: `resolution: found` (no action needed)
3. Annotations that no longer match above threshold: `resolution: orphaned`
4. Orphaned annotations are preserved with their original selector context
5. CI reports orphaned annotations as warnings (not errors)

**Tooling-generated annotations:**

- Generated annotations carry `creator: "{tool-name}"` and authority level `generated`
- They are treated as suggestions: visible in the editor but visually distinguished
- A human reviewer can promote a generated annotation by changing the creator to their own identity
- Conflicting generated annotations (two tools suggesting different linking targets for the same text) are both shown; the human picks the correct one

### 8. Editor integration

#### Read path

```
                                                  ┌─────────────────────┐
                                                  │   useAnnotations.js │
┌──────────────────┐    fetch    ┌────────────┐   │                     │
│ /data/annotations│───────────>│ annotations │──>│  WASM resolve()     │
│ /{lawId}/        │            │    .yaml    │   │         │           │
│ annotations.yaml │            └────────────┘   │         v           │
└──────────────────┘                              │  resolved positions │
                                                  └─────────┬───────────┘
                                                            │
                                                            v
                                                  ┌─────────────────────┐
                                                  │  AnnotatedText.vue  │
                                                  │  <mark> spans with  │
                                                  │  color by motivation│
                                                  └─────────────────────┘
```

**`frontend/src/composables/useAnnotations.js`:**

1. Fetches annotation YAML from `/data/annotations/{lawId}/annotations.yaml`
2. Calls `WasmEngine.resolveAnnotations()` to get match positions per article
3. Returns a reactive list of `{ annotation, match }` objects, filtered by the selected article

**`frontend/src/components/AnnotatedText.vue`:**

Replaces `ArticleText.vue` in the editor. Takes the article text and resolved annotation positions. Renders text with `<mark>` spans at the matched positions. Each span carries:

- Background color by motivation type (linking: blue, commenting: yellow, questioning: orange, tagging: green)
- Border style by authority level (authoritative: solid, advisory: dashed, generated: dotted)
- Click handler to show annotation details

#### Write path (MVP)

1. User selects text in `AnnotatedText.vue`
2. `useTextSelection.js` composable captures the selection, extracts `exact`, computes `prefix` (30-50 chars before) and `suffix` (30-50 chars after) from the article text
3. Validates uniqueness via `WasmEngine.resolveAnnotation()` — if the selector matches multiple locations, asks for more context
4. `AnnotationCreator.vue` opens as a popover: user picks motivation type, selects target `machine_readable` element (for linking), or types text (for commenting)
5. New annotation added to local state and persisted in `localStorage`
6. Export button downloads the updated annotation YAML for manual git commit

The right pane's segmented control gains a third option:

## Why

### Benefits

- **Git gives full annotation history for free.** Every annotation change is a commit with author, timestamp, and diff. `git blame` on an annotation file shows who added each annotation and when.

- **Sidecar files preserve verbatim legal text.** The law YAML is never modified by annotations (RFC-005 requirement). This is essential for legal integrity: the source text must be identical to the official publication.

- **Federated model lets every org bring their own annotations.** A municipality (*gemeente*) can annotate national laws from their perspective without modifying the central corpus. A legal expert can maintain personal annotations. This mirrors RFC-010's "bring your own regulations" model.

- **Authority model reflects legal reality.** The competent authority's (*bevoegd gezag*) interpretation carries more weight, just as in administrative law. The authority level is derived from existing schema fields (`competent_authority`), not configured separately — following RFC-009's principle that the law defines boundaries.

- **Linking annotations make interpretation auditable.** Every connection between text and `machine_readable` logic is an explicit, reviewable annotation. This supports the reasoning requirement (*motiveringsplicht*, AWB 3:46): the chain from law text to computation can be inspected by citizens, courts, and oversight bodies.

- **W3C standard enables interoperability.** The annotation format works with existing tools like Hypothesis, Apache Annotator, and Recogito. External parties can produce annotations in the same format without using the regelrecht editor.

- **Rust resolver is the single source of truth.** The same resolver code runs in the engine (for CI validation and server-side resolution) and in the browser (via WASM). No divergence between what the editor shows and what the engine validates.

### Tradeoffs

- **Two places to look.** A law's annotations live in a separate directory from its YAML source. Tooling must load both. The convention-based discovery makes this straightforward but it is an additional step.

- **No real-time collaboration.** Annotations are Git-based. Two people editing annotations simultaneously must coordinate via branches and pull requests. For the MVP this is acceptable; production use may require a collaboration layer.

- **WASM adds build complexity.** The WASM compilation step (`wasm-pack build`) is an additional build step. The engine already supports WASM compilation, so the incremental cost is low.

- **Authority derivation requires `competent_authority` in the law.** Laws without a declared `competent_authority` (*bevoegd gezag*) cannot distinguish authoritative from advisory annotations. This is an existing gap (RFC-009 also depends on `competent_authority` being declared).

### Alternatives Considered

**Alternative 1: Annotations inside law YAML**

Embed an `annotations` array directly in the law file.

Rejected: modifies the verbatim source text. RFC-005 explicitly requires annotations to be stand-off. Additionally, mixing annotations with law text in Git creates noisy diffs where annotation changes obscure law changes.

**Alternative 2: Database storage**

Store annotations in PostgreSQL (the pipeline already uses Postgres).

Rejected for MVP: introduces a backend dependency where none exists. The frontend is currently static. A database loses Git's built-in versioning, blame, and merge workflow. For production use with collaboration requirements, a database layer could be added later without changing the annotation format.

**Alternative 3: JavaScript-only resolver**

Implement TextQuoteSelector resolution directly in JavaScript.

Rejected: duplicates the fuzzy matching algorithm. The engine needs annotation resolution for CI validation (`just validate-annotations`); a separate JS implementation would diverge. WASM compiles the same Rust code for the browser.

**Alternative 4: Centralised annotation authority**

Designate a single organisation as the annotation authority for each law.

Rejected: doesn't match the federated model (RFC-010) or the multi-org execution reality (RFC-009). Multiple organisations have legitimate reasons to annotate the same law. The layering model accommodates this without requiring a single authority.

### Implementation Notes

Implementation is planned in six phases, each delivering a working whole.

**Phase 1 — Rust resolver + BDD tests**

Implement the `annotation` module with `types.rs` and `resolver.rs`. Port the 8 BDD scenarios from the `feature/annotation-resolver` branch. Add `strsim` dependency to `Cargo.toml`.

**Phase 2 — Annotation schema + first annotations**

Create `schema/v0.4.0/annotation-schema.json`. Write the first real linking annotations for the Healthcare Allowance Act (*Wet op de zorgtoeslag*) article 2. Add `validate-annotations` recipe to the Justfile.

**Phase 3 — WASM bindings**

Add `resolve_annotation()` and `resolve_annotations()` methods to `WasmEngine` in `wasm.rs`. Build and test with `wasm-pack`.

**Phase 4 — Frontend display**

Create `useAnnotations.js` composable and `AnnotatedText.vue` component. Integrate into `EditorApp.vue`. Update `copy-laws.js` build script to include annotation files.

**Phase 5 — Frontend creation**

Create `useTextSelection.js` composable and `AnnotationCreator.vue` component for creating linking annotations via text selection.

**Phase 6 — CI validation**

Add `just validate-annotations` to the quality gate. The recipe resolves all annotations against their target laws and reports orphaned annotations as warnings.
